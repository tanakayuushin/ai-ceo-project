## 1. CCR（Claude Code Remote）ベストプラクティス

### 効果的なプロンプト設計の鉄則

**悪いプロンプト例:**
```
「ファイルを読んで更新してください」
```

**良いプロンプト例:**
```
「以下の手順で実行してください:
1. departments/sales/見込み客リスト.md を読む
2. ステータスが「🆕 新規」の企業を抽出する
3. 各企業の最終更新日が7日以上前のものをフォローアップ対象とする
4. departments/sales/followup-list.md にリストを書き出す
5. git add → commit → push する

エラーが発生した場合は処理を中断し、エラー内容をファイルに記録すること。」
```

**ポイント:**
- ステップを番号で明示する
- 完了条件を具体的に書く
- エラー処理を指示に含める
- git操作（pull --rebase → push）を忘れずに含める

### CCRのgit操作テンプレート（必ずこの順序で）

```bash
git remote set-url origin https://<GITHUB_PAT>@github.com/tanakayuushin/ai-ceo-project.git
git pull --rebase origin main
# ... ファイル操作 ...
git add <files>
git commit -m "メッセージ (YYYY-MM-DD)"
git push origin main
```

### CCRのスケジュール設計パターン

| パターン | 用途 | cron例 |
|---------|------|--------|
| 毎朝定時実行 | ブリーフィング・レポート | `0 21 * * *` (JST 6:00) |
| 平日のみ | 営業フォローアップ | `0 0 * * 1-5` |
| 週次 | X投稿・週報 | `0 0 * * 1` |
| 月次 | 月次レポート・帳簿集計 | `0 0 1 * *` |

## 3. プロンプトキャッシュ（コスト削減）

### 仕組み

同じシステムプロンプト・文書を繰り返し使う場合、Anthropicのサーバーにキャッシュされ、2回目以降のコストが約90%削減される。

### キャッシュが効く条件

- キャッシュTTL: **5分間**（5分以内に同じプロンプトを使えばキャッシュヒット）
- 最小キャッシュサイズ: 1024トークン以上
- プロンプトの先頭部分が一致していること

### CCRでのキャッシュ活用

CCRの連続実行（毎朝のブリーフィング等）は同じシステムプロンプトを使うため、自然にキャッシュが効く設計にする。

## 4. Hooks活用（自動トリガー）

### 主要なHookイベント

| イベント | タイミング | 活用例 |
|---------|-----------|--------|
| `PreToolUse` | ツール実行前 | 危険なコマンドをブロック |
| `PostToolUse` | ツール実行後 | 実行ログを記録 |
| `Stop` | Claude応答終了時 | 完了通知を送信 |
| `SubagentStop` | サブエージェント終了時 | 結果を集約 |

### 設定例（settings.json）

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Bash executed' >> /tmp/claude-log.txt"
          }
        ]
      }
    ]
  }
}
```

## 6. Claude Codeの効率的な使い方（日常運用）

### コンテキスト管理

- **CLAUDE.md**: プロジェクト概要・ルール・ディレクトリ構成を記載（常に読み込まれる）
- **MEMORY.md**: 重要な記憶のインデックス（自動読み込み）
- **大きなファイルは分割**: 1ファイル2000行以内を目安に

### 効率的な指示の出し方

**悪い例:**「色々やっておいてください」

**良い例:**「以下の3つを今すぐ並列で実行してください:
1. ◯◯を調査して results.md に保存
2. ◯◯のメール下書きをGmailに作成
3. ◯◯のリストを更新してgit push」

### 定期的に行うべきメンテナンス

| 作業 | 頻度 | 内容 |
|------|------|------|
| メモリの整理 | 月1回 | 古い・不正確なメモリを削除・更新 |
| CCRトリガーの確認 | 週1回 | 実行ログを確認・失敗があれば修正 |
| .envのキーローテーション | 3ヶ月毎 | APIキーを再発行して.envを更新 |
| CLAUDE.mdの更新 | 変化があれば随時 | 組織変更・新サービス追加時に更新 |
